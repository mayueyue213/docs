这里还有两个关注点，先标记备忘，暂时不去深入：

  1. setActiveInstance(vm)的作用；
  2. HOC高阶组件的特殊处理；

这里pushTarget，popTarget等也是重要的内容，但是我们先暂时不去关注。

在实例的诸多方法中，只使用了get方法用以初始化getter，加上这里有run方法这个往往用作正式逻辑处理调用的命名，可以得出结论，这里的处理符合我们前面猜测的第2种，即 “缓存实例在其他地方调用”。根据这个实现，要使用到这个实例，则只要通过vm._watcher或vm._watchers，这可以作为切入点。

# 关于new一个实例

这里的Watcher实例化的方式是一个比较有意思的点，即new Watcher后并没有通过变量缓存再使用，借这个例子说一说JS的类与实例。

假设我们有这样一个Adder类

class Adder {
  nums = [];

  constructor(nums) {
    if (Array.isArray(nums)) {
      this.nums = nums;
    } else if (typeof nums === 'number') {
      this.nums.push(nums);
    }
  }

  run() {
    return this.nums.reduce((sum, val) => {
      return sum += val;
    }, 0);
  }
}

注意看看，这段代码如果像下面这样初始化会怎样

new Adder([1,2,3,4]);

你会发现，如果你不保存实例的引用，你无法在外部调用实例方法run，那么在这种初始化模式下，应该在内部会缓存了实例，比如说类似这样


class Adder {
  nums = [];

  constructor(target, nums) {
    // ...
    // 在这里将this缓存到某个参数上，或者某个其他属性上
    target._adder = this;
    // ...
  }

  run() {
    // ...
  }
}

在mount过程中new Watcher是下面这样的格式

new Watcher(vm, updateComponent, noop, {
  before () {
    if (vm._isMounted && !vm._isDestroyed) {
      callHook(vm, 'beforeUpdate')
    }
  }
}, true /* isRenderWatcher */)

使用这样的调用，则在Watcher的constructor的实现中就应该是会做实例缓存，如前面所展示的Watcher的代码所示，也确实如猜测的一样。

个人理解，实例是需要被使用的，应当先创建然后再通过实例方法去执行相应的操作，否则如果仅仅是执行了constructor就完事了，就相当于只是个普通的函数调用，却戴了个class的帽子，并不十分妥当。

总之，就是要么内部处理好，就像自给自足，要么就需要和外部进行沟通，这样子才能维持得了生活。
