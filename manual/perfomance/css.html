# CSS

CSS 怎么优化呢？于我而言，最直接想到的是从 文件大小 和 CSS对解析和渲染的影响 两个方面切入去思考。

# 更快 - 降低阻塞影响

CSS 会阻塞所有 JS 的执行，在 JS 之前的 CSS 会间接阻塞解析。根据这个特点，能做什么呢？考虑以下两种相对位置情况

<link rel="stylesheet" href="base.css">
<script src="tracker.js"></script>
1
2

以及这样

<script src="tracker.min.js"></script>
<link rel="stylesheet" href="base.css">
1
2

这看起来简单的区别，其实涉及到的知识非常复杂。判断两者的区别，需要结合以下知识

  • 浏览器的预加载能力
  • CSS，JS 加载各自对于解析的阻塞性
  • 重绘和回流的触发条件

只以 Chrome 为例说明，不考虑其他浏览器的差异。预加载能力的存在，script, CSS 文件的加载之间并不会相互阻塞。

第一种情况的大概处理流程(同级表示可并行，缩进表示顺序执行)

  • 并发加载 tracker.js 和 base.css
    • 等待 base.css 解析完成(因为 JS 要获取最新的样式)
      • tracker.js 执行

第二种情况的处理流程

  • 并发加载 tracker.js 和 base.css
    • 执行 tracker.js
    • 解析 base.css

所以理论上来说，第二种情况会更好。也许你会发现，这里可能需要考虑 tracker.js 是否依赖 base.css 应用后的样式？不过，如果是要应用在 head 里面的脚本，不就不需要考虑了么，因为 head 里面的 JS 本就不该操作 dom 影响可见内容的 body。 Google 首页就符合第二种情况

图中，第一个 JS 加载并执行之后，动态加载了后面的一个 JS，像这种情况，也能够使得第二个 JS 的加载进一步提前。

总结起来，就是在 head 里面，JS 资源可以考虑放到 CSS 资源前面。

# 加载优先级

按照前面的说法，由于 CSS 的阻塞性，而页面渲染对于 CSSOM 的依赖，FP(First Paint) 应该发生在任意 CSS 资源加载之后。 能不能设置 CSS 不阻塞渲染呢？有的，就像 <script> 有 async 属性能够让 JS 不阻塞渲染一样，<link> 有 media 属性能够让 CSS 资源不阻塞渲染。只不过相比 async/defer，这个 media 属性不那么为人所知所用。

<!DOCTYPE html>
<html lang="en">
<head>
  <title>Document</title>
  <link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/reset-css@5.0.1/reset.min.css"></link>
  <link rel="stylesheet" href="https://cdn.bootcdn.net/ajax/libs/nes.css/2.3.0/css/nes-core.css" media="(max-width: 375px;)"></link>
</head>
<body>
  <div id="app">demo</div>
</body>
</html>

当屏幕小于 375 时，与不使用 media 一样的策略，会阻塞 JS 执行，阻塞渲染。当屏幕大于 375 时，nes-core.css 的优先级会降低，更重要的是，不会阻塞 JS 执行，不会阻塞 FP。

总结起来，media 属性可用于对响应式的样式进行拆分，按照需要进行加载。

# 其他

  • 尽量避免使用代价昂贵的选择器和属性：我觉得这些稍微注意即可，像通配符，正则类，表达式一类的，很容易记住是代价比较高的，往往也有一些替代方案，但是像 border-radius，box-shadow 这些，不具备避免的可行性。

  • 关键 CSS 内嵌进入页面，异步加载其他非关键 CSS

# 总结与参考

CSS 的优化和 CSS 的掌握一样，都是看起来简单罢了。很多时候可以了解，但不一定要应用，花100分的力气优化的内容，其实对整体可能影响微乎其微，反而影响了开发体验，这就很不好了。
