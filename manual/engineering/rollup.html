# Rollup

Rollup 翻译过来是”汇总“，这体现了它的能力，将小段代码，模块，整合成一个更大更复杂的库或者应用。关于 Webpack 和 Rollup 的使用，有一句经典的话

Use webpack for apps, and Rollup for libraries

# 基本使用

命令行方式往往如下

rollup main.js -o bundle.js -f iife

使用配置文件

export default {
  input: 'src/main.js',
  output: [

# 插件

简单介绍几个插件，说明 Rollup 的插件能力，如果说 Webpack 还有 loader 进行处理，而 Rollup 就基本上全靠插件。

# @rollup/plugin-node-resolve

Rollup 把对于 npm 外部模块的依赖解析的功能放在了插件当中，并未”开箱即用“支持导入 npm 模块，需要增加 resolve 插件

import resolve from '@rollup/plugin-node-resolve';

export default {
  input: 'src/main.js',
  output: {
    file: 'bundle.js',
    format: 'cjs'
  },
  plugins: [ resolve() ]
};

# @rollup/plugin-commonjs

Webpack 开箱支持了多种 es, cjs 的混合使用，Rollup 没有，使用 cjs 模块需要通过插件先转换为 es

import commonjs from '@rollup/plugin-commonjs';

export default {
  input: 'src/index.js',
  output: {
    dir: 'output',
    format: 'cjs',
  },
  plugins: [commonjs()],
};

# 代码拆分

[!] Error: UMD and IIFE output formats are not supported for code-splitting builds.

在使用代码拆分时，是不支持指定 IIFE 或者 UMD 格式的代码拆分的，只支持 ES Module 和 CommonJS。

而对于 ES Module，由于兼容性问题，在浏览器端使用时，需要对输出的内容做进一步处理。

# 总结

Webpack 对于代码分割，静态资源有着优势，而 Rollup 更加轻量，配置更加简洁，能够方便地将各模块汇总，打包成各种格式的输出。至于 Tree-Shaking，虽然是由 Rollup 引入，但现在都已经支持了，也算不得 Rollup 的优势了。

在实际使用中，我的选择也基本上是 Use webpack for apps, and Rollup for libraries。
