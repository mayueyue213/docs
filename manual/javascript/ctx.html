# 执行上下文

执行上下文(Execution Contexts，简称EC)是执行一段代码时的运行环境，也可以理解为这段代码被执行时的状态描述。理解”一段“的概念很重要，这个”一段“是管理区间的划分，对应执行上下文的分类

  • 全局执行上下文
  • 函数执行上下文
  • eval执行上下文

仔细思索一下，全局，函数，eval，有什么共同点？

其内部代码，在其执行时才会进行编译

根据 ECMA规范中的定义 (opens new window)，上下文由以下三个部分组成

变量环境和 this 可以当做对象去看待，this 对象，变量环境对象，而词法环境应该当做是一个栈去理解，词法环境栈，栈中的每一项称作词法环境对象。

# 变量环境

Identifies the Lexical Environment whose environment record holds bindings created by VariableStatements and FunctionDeclarations within this execution context.

# 词法环境

Identifies the Lexical Environment used to resolve identifier references made by code within this execution context.

这个描述并不容易理解，尤其是 within this execution context。变量环境和执行上下文是一一对应，可以这样描述，但词法环境不与上下文一一对应，而是与词法作用域对应，这种描述就不是很便于理解了。可以简单理解为

每个词法作用域对应有一个词法环境对象，这个词法环境对象上保存了其中定义的变量。

根据这个解释，函数作用域，块级作用域，都是一个词法作用域，则

  • {}局部作用域，对应一个词法环境对象，保存其中定义的变量
  • 函数局部作用域，对应一个词法环境对象，保存其中定义的变量

但是由于 作用域嵌套 和 变量提升 的存在，事情变得不那么简单了

  • 变量提升：由于变量提升，d的作用范围并不局限于{}。通过 var 声明的变量，是属于执行上下文级别的，而不是局部词法环境，而 词法环境对象上只保存了使用 let, const 声明的局部变量。
  • 作用域嵌套：foo 对应一个词法环境对象，{}对应一个词法环境对象，由于嵌套，需要对词法环境对象间的关系进行管理，并且由于{}对应的词法环境对象，只有在代码执行到其内部才有意义，因此执行上下文中引入了词法环境栈来管理嵌套的词法环境(作用域)。局部作用域就是通过这样来实现的呀。

  • 在局部作用域 {} 中通过let,const声明的变量，保存在了一个单独的词法环境对象中，并入栈。

# 变量对象

《JavaScript高级教程》里面对“变量对象”(Variable Object，简称VO)是这样描述的：

每个上下文都有一个关联的变量对象（variable object），而这个上下文中定义的 所有变量和函数 都存在于这个对象上

在没有let和const的年代，上面的描述可以等价于

每个上下文都有一个关联的变量对象（variable object），而这个上下文中 var 定义的所有变量，函数声明 都存在于这个对象上

当加入了 let 和 const，加入了块级作用域之后，这两个描述不等价了。

原来的”变量对象“就是如今的“变量环境”，如果仍要使用 “变量对象” 这个概念，那么应该仍旧按照第一个描述，仍旧是描述一个“保存上下文中定义的所有变量和函数”的对象。

# 总结

在引入块级作用域之前，用一个对象去保存所有变量的引用就可以了，这在以前叫做变量对象。引入块级作用域之后，由于普通块级作用域执行并不创建新的上下文，通过将变量对象拆分，用变量环境来替代原变量对象，用词法环境来隔离嵌套作用域中的同名变量，通过词法环境栈来维护变量查找顺序。

在执行上下文中，对于 “保存定义的变量和函数的对象”，有几个名词：

名词 缩写 含义
变量对象 VO 保存上下文中定义的所有变量的对象
变量环境 VE 保存上下文中定义的所有 var 声明 变量和函数标识符的对象
词法环境 LE 保存作用域中定义的所有 let, const 声明 变量和函数标识符的对象
